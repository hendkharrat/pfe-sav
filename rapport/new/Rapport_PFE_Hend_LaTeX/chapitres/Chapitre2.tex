Après avoir présenté le cadre général du projet, nous entamons la phase de préparation du projet
selon la méthodologie Scrum. Cette étape vise à identifier, analyser et modéliser les besoins du

\section{Capture des besoins}

\subsection{Spécification des besoins}

\subsubsection{Exigences fonctionnelles}

L'application doit permettre d'assurer les fonctionnalités suivantes :

\textbf{1. Authentification et gestion des sessions}
\begin{itemize}
    \item Connexion par e-mail ou numéro de téléphone avec mot de passe.
    \item Gestion des accès selon les rôles (Administrateur, Technicien, Client).
    \item Déconnexion sécurisée.
\end{itemize}

\textbf{2. Gestion des utilisateurs internes}
\begin{itemize}
    \item Création, consultation, modification et désactivation des utilisateurs.
    \item Réactivation des comptes désactivés.
    \item Gestion des administrateurs et techniciens.
\end{itemize}

\textbf{3. Gestion des clients}
\begin{itemize}
    \item Création, consultation, modification et suppression des clients.
    \item Gestion des clients société ou personne physique.
    \item Sélection de la ville depuis une liste tunisienne.
    \item Accès client via mot de passe.
\end{itemize}

\textbf{4. Gestion des équipements}
\begin{itemize}
    \item Gestion d'un catalogue global d'équipements.
    \item Ajout, modification, consultation et suppression des équipements.
    \item Gestion des types d'équipements (climatiseur ou système de surpression).
    \item Gestion de plusieurs images par équipement avec image principale.
\end{itemize}

\textbf{5. Affectation des équipements aux clients}
\begin{itemize}
    \item Association d'un équipement à un client.
    \item Enregistrement de la date d'achat, de la date d'installation et de la localisation.
    \item Affectation depuis le module Client ou Équipement.
\end{itemize}

\textbf{6. Gestion des contrats}
\begin{itemize}
    \item Création, consultation, modification et suppression des contrats.
    \item Association d'un contrat à un client et à ses équipements.
    \item Définition de la périodicité (mensuelle, trimestrielle, semestrielle ou annuelle).
    \item Calcul automatique du statut du contrat.
\end{itemize}

\textbf{7. Génération du planning préventif}
\begin{itemize}
    \item Génération automatique des interventions préventives.
    \item Prévisualisation du planning avant validation du contrat.
\end{itemize}

\textbf{8. Gestion des interventions}
\begin{itemize}
    \item Gestion des interventions préventives et curatives.
    \item Affectation des techniciens.
    \item Vérification automatique de la disponibilité des techniciens.
\end{itemize}

\textbf{9. Gestion des pannes}
\begin{itemize}
    \item Déclaration des pannes par le client.
    \item Ajout d'une description et de pièces jointes.
    \item Conversion d'une \texttt{Panne} en intervention curative par l'administrateur.
\end{itemize}

\textbf{10. Gestion de la facturation}
\begin{itemize}
    \item Génération des factures pour les interventions curatives hors contrat.
    \item Calcul du montant HT, de la TVA (19\,\%) et du montant TTC.
    \item Marquage des factures comme payées.
\end{itemize}

\textbf{11. Tableaux de bord}
\begin{itemize}
    \item Tableau de bord global pour l'administrateur.
    \item Tableau de bord personnel pour le technicien.
    \item Espace personnel pour le client.
\end{itemize}

\textbf{12. Historique et planning}
\begin{itemize}
    \item Consultation de l'historique selon le rôle.
    \item Consultation du planning en vue hebdomadaire ou mensuelle.
\end{itemize}

\subsubsection{Exigences non fonctionnelles}

Le système doit également respecter plusieurs contraintes non fonctionnelles :

\textbf{Sécurité} : authentification obligatoire, contrôle d'accès basé sur les rôles (RBAC),
session sécurisée sans stockage du mot de passe, chiffrement des mots de passe en base de données.

\textbf{Performance} : temps de réponse rapide, listes paginées pour optimiser les performances.

\textbf{Ergonomie} : interface entièrement en français, intuitive, responsive (desktop, tablette,
mobile), thème clair et thème sombre.

\textbf{Maintenabilité} : code TypeScript avec typage statique strict, architecture modulaire,
composants réutilisables, séparation des responsabilités.

\textbf{Disponibilité} : accessibilité à tout moment depuis un navigateur web moderne, sans
installation préalable.

\textbf{Localisation} : monnaie TND, TVA 19\,\%, villes tunisiennes.

\textbf{Fiabilité} : validation des données saisies côté client, messages d'erreur explicites.

\textbf{Relations entre les entités :}

Les relations entre les entités du dictionnaire de données sont les suivantes :

\begin{itemize}
    \item \texttt{Client} $1 \rightarrow *$ \texttt{ClientEquipement}
    \item \texttt{Equipment} $1 \rightarrow *$ \texttt{ClientEquipement}
    \item \texttt{Equipment} $1 \rightarrow *$ \texttt{EquipmentImage}
    \item \texttt{Contract} couvre $*$ \texttt{ClientEquipement} (via \texttt{clientEquipementIds[]}) ---
          un contrat peut couvrir plusieurs installations \texttt{ClientEquipement} ;
          une installation peut être couverte par plusieurs contrats dans le temps,
          mais par un seul contrat actif à la fois
    \item \texttt{Intervention} $* \rightarrow 1$ \texttt{ClientEquipement}
    \item \texttt{Intervention} $* \rightarrow 0..1$ \texttt{Contract}
    \item \texttt{Panne} $1 \rightarrow *$ \texttt{PieceJointe}
    \item \texttt{Panne} $0..1 \rightarrow 1$ \texttt{Intervention} (conversion)
    \item \texttt{Facture} $* \rightarrow 1$ \texttt{Intervention}
    \item \texttt{Facture} $1 \rightarrow *$ \texttt{LigneFacture}
    \item \texttt{Intervention} $* \rightarrow 0..1$ \texttt{User} (technicien affecté)
\end{itemize}

\section{Architecture de l'application}

L'application est construite sur une architecture web en trois couches qui sépare clairement le
frontend, la logique métier et la base de données.

\textbf{Couche présentation (Frontend)} Le frontend de l'application est développé en utilisant
Next.js et React. Cette combinaison permet de créer une interface utilisateur à la fois moderne
et dynamique. Les composants React assurent une séparation claire des responsabilités d'affichage.

\textbf{Couche logique métier (Backend / API)} La couche backend expose des routes API RESTful
implémentées sous forme de contrôleurs. Chaque contrôleur gère la validation des données,
l'exécution de la logique métier et les appels vers la base de données.

\textbf{Couche d'accès aux données (Prisma / MySQL)} L'accès à la base de données est géré
via Prisma ORM. Cette couche exécute les requêtes vers la base de données MySQL et garantit
l'intégrité des données via la gestion des transactions (COMMIT/ROLLBACK).

\section*{Conclusion}

Dans ce chapitre, nous avons établi les exigences fonctionnelles et non fonctionnelles de
l'application en précisant leurs règles métier. Nous avons repéré quatre protagonistes
(Administrateur, Technicien, Client, Système) et représenté leurs interactions à travers 28
User Stories réparties sur 3 sprints. Le schéma de classes global présente les 11 entités du
modèle, incluant l'entité intermédiaire \texttt{ClientEquipement} qui associe les clients aux
équipements du catalogue. Le prochain chapitre traitera du Sprint 1 --- Authentification et
référentiel métier.
