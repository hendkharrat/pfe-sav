Dans le chapitre précédent, nous avons traité de la phase préparatoire du projet ainsi que des
spécifications du système. Ce chapitre aborde le premier sprint du projet, réalisé selon la méthode
Scrum. Ce sprint est essentiel pour le développement de l'application, car il établit les fondations
du système, surtout en ce qui concerne l'authentification des utilisateurs et la gestion des données
cruciales.

Les fonctionnalités développées pendant ce sprint sont principalement axées sur :

\begin{itemize}
    \item L'authentification par e-mail ou numéro de téléphone et la déconnexion des utilisateurs ;
    \item La gestion des utilisateurs internes avec désactivation logique et restauration ;
    \item La gestion des clients (société ou personne physique) ;
    \item La gestion du catalogue d'équipements avec images multiples ;
    \item L'affectation des équipements aux clients via \texttt{ClientEquipement} ;
    \item La gestion des contrats de maintenance avec génération automatique du planning préventif.
\end{itemize}

    \item Modifier un utilisateur existant ;
    \item Désactiver un utilisateur (soft-delete : \texttt{actif = false}) ;

\textbf{Scénario nominal (Créer) :}
\begin{enumerate}
    \item L'administrateur saisit les informations du nouvel utilisateur ainsi que son rôle.
    \item L'interface transmet les données au contrôleur.
    \item Le contrôleur valide le format de l'e-mail et du téléphone.
    \item Le contrôleur vérifie l'unicité de l'e-mail.
    \item La base de données confirme que l'e-mail n'existe pas.
    \item Le contrôleur chiffre le mot de passe.
    \item Le contrôleur demande l'enregistrement du nouvel utilisateur.
    \item La base de données enregistre les données.
    \item La base confirme l'insertion ; la liste des utilisateurs est mise à jour.
\end{enumerate}

\textbf{Scénario nominal (Désactiver) :}
\begin{enumerate}
    \item L'administrateur clique sur « Désactiver » pour un utilisateur actif.
    \item Le système effectue une mise à jour : \texttt{actif = false}.
    \item L'utilisateur disparaît des listes actives mais reste visible dans la liste complète
          avec une action « Restaurer ».
\end{enumerate}

\textbf{Règles de gestion :}
\begin{itemize}
    \item Chaque utilisateur interne doit avoir un e-mail unique.
    \item Le mot de passe est chiffré avant stockage.
    \item Un utilisateur désactivé (\texttt{actif = false}) ne peut plus se connecter.
    \item L'action « Restaurer » remet \texttt{actif = true}.
    \item Les comptes clients ne sont pas gérés ici --- ils appartiennent au module Clients.
\end{itemize}

\hline
\textbf{Acteur principal}
  & Administrateur \\
\hline
\textbf{Préconditions}
  & L'administrateur est authentifié ; le client existe ; des enregistrements

  & Le contrat est enregistré en base ; les interventions préventives sont générées

\hline
\end{tabular}

\textbf{Scénario nominal :}
\begin{enumerate}
    \item L'administrateur saisit les informations du contrat : client, date de début, date de fin,
          périodicité, installations \texttt{ClientEquipement} couvertes.
    \item L'interface transmet les données au contrôleur.
    \item Le contrôleur vérifie que la date de fin est supérieure à la date de début.
    \item Le contrôleur vérifie que les \texttt{ClientEquipement} sélectionnés appartiennent
          bien au client choisi.
    \item Le système démarre une transaction.
    \item La base de données enregistre le contrat avec la liste \texttt{clientEquipementIds[]}.
    \item Le \textbf{Système} génère automatiquement les interventions préventives selon
          la périodicité et les installations couvertes.
    \item Le système valide la transaction (COMMIT).
    \item L'interface affiche la confirmation de création.
\end{enumerate}

\textbf{Scénarios alternatifs :}
\begin{itemize}
    \item Dates invalides (fin $\leq$ début) $\rightarrow$ message d'erreur affiché.
    \item \texttt{ClientEquipement} non rattachés au client $\rightarrow$ erreur
          « installations hors client ».
    \item Erreur base de données $\rightarrow$ transaction annulée (ROLLBACK) ;
          message d'erreur affiché.
\end{itemize}

\textbf{Règles de gestion :}
\begin{itemize}
    \item Un contrat couvre des installations \texttt{ClientEquipement}, et non des équipements
          du catalogue directement.
    \item Le statut est calculé dynamiquement : \texttt{ACTIF} si fin > aujourd'hui + 30 jours ;
          \texttt{BIENTOT\_EXPIRE} si fin entre aujourd'hui et aujourd'hui + 30 jours ;
          \texttt{EXPIRE} si fin < aujourd'hui.
    \item La génération du planning est automatique et ne requiert aucune action manuelle
          de l'administrateur.
\end{itemize}

    \item Le Système calcule les dates des interventions selon la périodicité choisie.
    \item Pour chaque date calculée et chaque \texttt{ClientEquipement} couvert, le Système
          crée une intervention préventive.
    \item Le Système enregistre toutes les interventions dans la base de données.
    \item La prévisualisation du planning est affichée à l'administrateur avant la confirmation
          finale.
\end{enumerate}

\textbf{Règles de gestion :}
\begin{itemize}
    \item La génération est automatique : elle est déclenchée par la sauvegarde du contrat,
          non par une action manuelle.
    \item Une intervention préventive est générée pour chaque couple
          (date, \texttt{ClientEquipement} couvert).
    \item Périodicités : mensuelle (12/an), trimestrielle (4/an), semestrielle (2/an),
          annuelle (1/an).
\end{itemize}

\textbf{Identification textuelle --- Connexion :}

Nom : S'authentifier. Acteur principal : Tout utilisateur (Admin / Technicien / Client).
Participants : Interface (Frontend) $\rightarrow$ Contrôleur (AuthController)
$\rightarrow$ Base de données (MySQL).

Scénario nominal :
\begin{enumerate}
    \item L'utilisateur saisit son identifiant et son mot de passe.
    \item L'Interface transmet les données au Contrôleur.
    \item Le Contrôleur détecte le type d'identifiant :
    \begin{itemize}
        \item Si l'identifiant contient un « @ » $\rightarrow$ recherche par e-mail
              dans \texttt{User} puis \texttt{Client}.
        \item Sinon (8 chiffres) $\rightarrow$ recherche par téléphone dans \texttt{User}
              puis \texttt{Client}.
    \end{itemize}
    \item Le contrôleur interroge la base de données.
    \item La base retourne l'enregistrement correspondant (ou null si non trouvé).
    \item Le Contrôleur vérifie le mot de passe (hash comparison).
    \item Si valide et compte actif $\rightarrow$ le Contrôleur crée la session et retourne
          les informations de l'utilisateur.
    \item L'Interface redirige l'utilisateur vers son tableau de bord selon le rôle.
\end{enumerate}

Scénarios alternatifs :

\subsubsection{Identification textuelle --- Créer un utilisateur}

Nom du cas d'utilisation : Créer un utilisateur. Acteur principal : Administrateur.
Objectif : Permettre à l'administrateur d'ajouter un nouvel utilisateur interne dans le système
avec un rôle spécifique.

Préconditions :
\begin{itemize}
    \item L'administrateur est authentifié.
    \item La connexion à la base de données est disponible.
\end{itemize}

Postconditions :
\begin{itemize}
    \item Le nouvel utilisateur est enregistré dans la base de données.
    \item La liste des utilisateurs est mise à jour.
\end{itemize}

Scénario nominal :
\begin{enumerate}
    \item L'administrateur saisit les informations du nouvel utilisateur ainsi que son rôle.
    \item L'Interface transmet les données au Contrôleur.
    \item Le Contrôleur valide le format de l'e-mail et du numéro de téléphone.
    \item Le Contrôleur demande la vérification de l'unicité de l'e-mail.
    \item Le contrôleur interroge la base de données.
    \item La base retourne le résultat de la recherche.
    \item Si l'e-mail n'existe pas, le Contrôleur chiffre le mot de passe.
    \item Le Contrôleur demande l'enregistrement du nouvel utilisateur.
    \item La base de données enregistre les données.
    \item La base confirme l'enregistrement.
    \item Le système rafraîchit la liste des utilisateurs.
    \item L'Interface affiche la liste mise à jour à l'administrateur.
\end{enumerate}

Scénario alternatif 1 --- E-mail déjà utilisé :
\begin{itemize}
    \item La base de données indique que l'e-mail existe déjà.
    \item Le Contrôleur envoie un message d'erreur à l'Interface.
    \item L'Interface affiche le message « E-mail déjà utilisé ».
\end{itemize}

Scénario alternatif 2 --- Erreur base de données :
\begin{itemize}
    \item Une erreur survient pendant l'insertion.
    \item La base de données retourne une erreur au contrôleur.
    \item L'Interface affiche un message d'erreur à l'administrateur.
\end{itemize}

Règles de gestion :
\begin{itemize}
    \item Chaque utilisateur interne doit avoir un e-mail unique.
    \item Le mot de passe doit être chiffré avant l'enregistrement.
    \item Le rôle doit appartenir aux rôles autorisés : admin ou technician.
    \item Le numéro de téléphone est stocké sur 8 chiffres sans indicatif.
\end{itemize}

\subsubsection{Identification textuelle --- Suppression utilisateur (soft-delete)}

Nom du cas d'utilisation : Désactiver un utilisateur. Acteur principal : Administrateur.
Objectif : Permettre à l'administrateur de retirer l'accès d'un utilisateur sans supprimer
définitivement ses données (soft-delete).

Préconditions :
\begin{itemize}
    \item L'administrateur est authentifié.
    \item L'utilisateur à désactiver existe et son \texttt{actif} est \texttt{true}.
\end{itemize}

Postconditions :
\begin{itemize}
    \item L'utilisateur est désactivé (\texttt{actif = false}).
    \item L'utilisateur ne peut plus se connecter.
    \item L'utilisateur reste visible dans la liste avec une action « Restaurer ».
\end{itemize}

Scénario nominal :
\begin{enumerate}
    \item L'administrateur clique sur « Désactiver » dans la ligne de l'utilisateur.
    \item L'Interface envoie la demande avec l'identifiant de l'utilisateur au Contrôleur.
    \item Le Contrôleur vérifie que l'utilisateur existe et est actif.
    \item Le contrôleur met à jour la base de données.
    \item La base de données effectue la mise à jour : \texttt{actif = false}.
    \item La base de données confirme la mise à jour.
    \item L'Interface met à jour la liste et affiche le bouton « Restaurer »
          pour cet utilisateur.
\end{enumerate}

Scénario alternatif --- Erreur base de données :
\begin{itemize}
    \item Une erreur survient lors de la mise à jour.
    \item La base de données retourne une erreur au contrôleur.
    \item L'Interface affiche un message d'erreur à l'administrateur.
\end{itemize}

Règles de gestion :
\begin{itemize}
    \item La désactivation est logique : \texttt{actif = false}.
    \item Les utilisateurs désactivés ne peuvent pas se connecter au système.
    \item Les données historiques de l'utilisateur sont conservées dans la base de données.
    \item L'action « Restaurer » remet \texttt{actif = true} et permet à l'utilisateur
          de se reconnecter.
\end{itemize}

\subsubsection{Identification textuelle --- Créer un contrat et générer le planning}

Nom du cas d'utilisation : Créer un contrat de maintenance. Acteur principal : Administrateur.
Objectif : Permettre à l'administrateur de créer un contrat de maintenance et déclencher la
génération automatique du planning préventif par le Système.

Préconditions :
\begin{itemize}
    \item L'administrateur est authentifié.
    \item Le client existe dans la base de données.
    \item Des enregistrements \texttt{ClientEquipement} existent pour ce client.
\end{itemize}

Postconditions :
\begin{itemize}
    \item Le contrat est enregistré dans la base de données avec ses
          \texttt{clientEquipementIds[]}.
    \item Les interventions préventives sont générées automatiquement par le Système.
    \item La transaction est validée avec succès.
\end{itemize}

Scénario nominal :
\begin{enumerate}
    \item L'administrateur saisit les informations du contrat et sélectionne les installations
          \texttt{ClientEquipement} couvertes.
    \item L'Interface transmet les données au Contrôleur.
    \item Le Contrôleur valide les dates du contrat (fin > début).
    \item Le Contrôleur vérifie que les \texttt{ClientEquipement} appartiennent au client
          sélectionné.
    \item Le système démarre une transaction.
    \item La base de données enregistre le contrat.
    \item Le \textbf{Système} analyse les \texttt{ClientEquipement} couverts et la périodicité.
    \item Le Système calcule les dates d'intervention selon la périodicité.
    \item Le Système crée une intervention préventive pour chaque couple
          (date, \texttt{ClientEquipement}).
    \item La base de données enregistre toutes les interventions générées.
    \item Le système valide la transaction (COMMIT).
    \item L'Interface affiche la confirmation et le planning généré.
\end{enumerate}

Scénarios alternatifs :
\begin{itemize}
    \item SA1 : Dates invalides (fin $\leq$ début) $\rightarrow$ message d'erreur ;
          pas de création.
    \item SA2 : \texttt{ClientEquipement} hors client $\rightarrow$ erreur
          « installations hors client ».
    \item SA3 : Erreur base de données $\rightarrow$ ROLLBACK ; message d'erreur.
\end{itemize}

Règles de gestion :
\begin{itemize}
    \item Un contrat couvre des installations \texttt{ClientEquipement}, non des équipements
          du catalogue directement.
    \item La génération est automatique ; l'administrateur n'a pas à lancer manuellement
          cette opération.
    \item Les opérations d'enregistrement sont exécutées dans une transaction atomique.
\end{itemize}
